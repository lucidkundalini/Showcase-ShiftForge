# Origin And Intended Architecture

## Scope

This note distills important parts of the older `backup_branch` README into a public, truthful form.

It separates:

- what ShiftForge was originally trying to become
- what the current live path actually proves
- what remains non-authoritative or unfinished

## What The Backup Branch Clearly Intended

The older branch consistently described ShiftForge as more than a single chat surface.

[certain] The intent in that branch was:

- a multi-agent orchestration system rather than one static prompt wrapper
- a loop of perception, routing, action, reflection, and learning
- model selection based on task type instead of a single fixed provider
- explicit safety layers around execution, memory, and tool use
- a long-term goal of turning software work into an inspectable operating system

Those themes are directly visible in the older README language and in the module names that still exist in the private tree.

## What Is Still Publicly Supportable Today

[certain] The current live product path still supports these claims:

- ShiftForge is an orchestration layer, not just a raw model label
- runtime truth answers are generated from runtime metadata for key identity/runtime questions
- trace ids and inspectable diagnostics are part of the response path
- provider and MCP introspection can be exposed in a bounded, secret-safe way
- the project contains explicit routing, observability, governance, and truth-layer code

## What I Am Not Re-Publishing As Fact

[certain] The older branch also claimed a much broader cognitive and model network surface.

[likely] Parts of that vision influenced the current private tree.

[guessing] I do not have enough current public evidence to present all of the following as live product truth:

- the full historical model roster from the old README
- every named cognitive layer as an active production component
- self-learning claims beyond the verified trace/governance/runtime surfaces
- every dormant or partially wired module as currently authoritative

Those items may exist in code history or private branches, but this showcase does not elevate them into current verified status without live proof.

## Public Interpretation

The backup branch matters because it shows that ShiftForge was designed around:

1. orchestration
2. bounded execution
3. model/provider routing
4. inspectability
5. iterative hardening

That intention is still visible in the current architecture. What changed is the standard of truth: this showcase only publishes the parts that can be defended with live code and runtime evidence.
